Opening Statement

Read this aloud to open your presentation.

Read aloud In August 2017, Kenya's Supreme Court did something unprecedented in African history: it nullified a presidential election result. The reason was not fraud at the ballot box — it was a compromised digital transmission system. Christopher Msando, the IEBC official responsible for that system, was found dead before the election.

One month earlier in 2007, disputed election results triggered violence that killed 1,100 Kenyans, displaced 600,000, and erased $3.6 billion from the economy.

These are not hypothetical risks. They are Kenya's recent history.

VeriVote Kenya is a full-stack, production-architecture e-voting system that makes those categories of attack mathematically impossible. Votes are encrypted before they leave the voter's browser. Every encrypted vote is anchored on an immutable blockchain. Any voter — and any international observer — can verify their vote was correctly recorded without revealing how they voted. And when a voter is being coerced, the system silently alerts authorities while showing the attacker a normal vote confirmation.

I built this over 8 weeks. Let me show you how.

System Design — How It Works

Use this section to walk judges through the architecture in plain language before diving into milestones.

The One Core Principle The entire system is built on one idea: you can prove a vote was cast and counted correctly, without anyone being able to see who you voted for. Your vote is locked inside a mathematical puzzle (ElGamal encryption) before it ever leaves your device. Only one key in the world can unlock it — held by the IEBC — and it is only used once, at the end of the election, to count everything simultaneously. No individual vote is ever revealed; just the totals.

Feature 1 — Voter Identity: "Are You Really You?"

Three independent layers must all pass before a vote can be cast:

1KYC — Government ID Verification

Before registering, the voter uploads their Kenyan national ID through Persona (government-grade identity verification). It checks that the ID is genuine and that the selfie matches the photo. A unique personaId is bound to the account — registering twice with the same ID is impossible.

2WebAuthn — Biometric Key (Fingerprint / Face)

After KYC, the voter registers their phone's biometric. The device generates a cryptographic key pair — the private key never leaves the phone's secure chip. At vote time, the biometric signs a server challenge with that key. The server verifies the signature. No password is ever transmitted over the network.

3PIN — Last Line of Defence

A 4-digit PIN (not 1111, not sequential) is required at vote submission. Guards against someone using an unlocked phone.

Distress PIN: A second PIN is auto-generated and sent secretly. If the voter is being coerced, they enter the distress PIN instead. The vote goes through normally — the coercer sees a confirmation — but the system silently flags it and fires SMS alerts to every IEBC coordinator.

Feature 2 — Vote Encryption: "Locking the Ballot Box"

When the voter taps Submit, this happens on their device before anything is sent:

Your selections:  { president: "Amina Wanjiku", governor: "Grace Muthoni" }
          ↓
  Encrypted with IEBC public key  (ElGamal 2048-bit FFDHE, RFC 7919)
          ↓
Ciphertext:  { president: { c1: "84f2a3…", c2: "9c71b8…" }, governor: { … } }

The ciphertext is what gets stored in the database. Even if the entire database were leaked, no one could read any vote without the private key. The key is only used once — during the decryption ceremony after polls close.

Feature 3 — Blockchain Anchoring: "The Tamper-Proof Ledger"

When the encrypted vote is saved, a fingerprint of it is written immediately to an Ethereum smart contract running on IEBC infrastructure:

SHA-256(encrypted_ciphertext)  →  64-char hex string
        ↓
keccak256(that string)  →  bytes32 stored on-chain forever

The blockchain is append-only — records can never be edited or deleted. If anyone later alters the database row, the fingerprint will no longer match the on-chain record. The tampering is mathematically detectable. What is NOT stored on-chain: selections, ciphertext, or any voter identifier — only the fingerprint.

Feature 4 — Re-Voting: "You Can Change Your Mind"

Voters may re-cast to counter coercion (attacker forces a vote, voter returns later to change it). Only the last vote counts:

LayerWhat happens on re-vote
DatabaseOld vote row → status = SUPERSEDED (kept forever as audit). New vote row created under a fresh serial number.
BlockchainsupersedeVote(oldSerial, newSerial, newHash) atomically marks the old fingerprint void and writes the new one. Both records stay on-chain permanently.
Decryption ceremonyOnly tallies WHERE status = CONFIRMED — superseded votes are skipped automatically.

Feature 5 — Paper Receipts: "Physical Safety Net"

After a vote is cast, it enters a PrintQueue. An IEBC-controlled printer at the station claims the job and prints a paper receipt containing the serial number and a QR code (serial + hash). The voter keeps this slip and can scan it on the public portal to verify their vote appears in the blockchain — without revealing who they voted for.

At the end of the election the decryption ceremony cross-checks: Digital votes decrypted =?= Physical receipts printed. Any discrepancy flags a DISCREPANCY alert.

Feature 6 — Decryption Ceremony: "Counting Day"

An IEBC administrator triggers the ceremony from the admin dashboard. The system:

  1. Fetches all CONFIRMED votes from PostgreSQL
  2. Decrypts each with the private key: m = c2 × (c1x)−1 mod p
  3. Tallies selections per candidate per position
  4. Computes a SHA-256 hash of the canonical results JSON
  5. Publishes that hash on-chain as an immutable seal — any future tampering with the published results would produce a different hash

A full ceremony log shows every decrypted vote (serial, station, selections), timing, turnout, and the final hash. Downloadable as a signed audit JSON.

Feature 7 — AI Fraud Detection: "The Watchman"

A separate Python microservice (FastAPI, port 8000) runs entirely on IEBC infrastructure. It uses a Random Forest model to detect anomalous voting patterns and a local LLM (Ollama-served Llama 3) to generate human-readable explanations of flagged anomalies. No data leaves Kenyan infrastructure. No calls to OpenAI, Google, or any foreign API.

Feature 8 — Real-Time Dashboard

Live vote counts update via WebSocket (Socket.io) as votes come in. The public portal shows live turnout, votes-per-hour chart, the blockchain explorer (last 20 confirmed fingerprints), and county-by-county breakdown. Distress alerts are visible to admins only.

What VeriVote cannot do — by design It cannot reveal who a voter chose (encryption is one-way until the ceremony). It cannot delete a blockchain record (chain is append-only). It cannot count votes without the private key (the key is the only path from ciphertext to plaintext). It cannot hide fraud (AI, reconciliation check, and on-chain seal all independently verify integrity).

SoulBoundToken.sol — non-transferable voter identity

smart-contracts/contracts/SoulBoundToken.sol 
// All transfer functions blocked at the EVM level
function transferFrom(address, address, uint256) public pure override {
    revert("SBT: non-transferable");
}
function approve(address, uint256) public pure override {
    revert("SBT: non-transferable");
}

// One SBT per national ID hash — mathematically enforced
function mint(address to, uint256 nationalIdHash)
    external onlyOwner returns (uint256)
{
    require(_idHashToToken[nationalIdHash] == 0,
            "SBT: already minted for this ID");
    require(!_hasActiveToken[to],
            "SBT: address already has token");
    // mints and records bidirectional mappings
}

Why SBTs? An ERC721 NFT can be sold or transferred. A Soul-Bound Token cannot. Once issued, it is permanently bound to that national ID hash at the contract level. One citizen. One immovable digital credential.

Milestone 4 — Encryption & Vote Verification Week 4

Rationale The key question: how do you let a voter prove their vote was recorded, without revealing what they voted for — to anyone, including the system itself? The answer is two layers. ElGamal probabilistic encryption makes the ciphertext unintelligible. A SHA-256 hash of that ciphertext acts as a fingerprint — you can verify the ciphertext hasn't been tampered with without knowing what it contains. That hash goes on the blockchain. The voter's serial number is the lookup key.

What was built

  • Full ElGamal encryption over a 2048-bit FFDHE group (RFC 7919 standard)
  • SHA-256 hash of every encrypted vote stored alongside the ciphertext
  • GET /api/votes/verify/:serial — recomputes hash, queries blockchain, tri-state result
  • /verify web page — public, bilingual (English/Kiswahili), printable

ElGamal probabilistic encryption

backend/src/services/encryption.service.ts — encrypt() 
// encryption.service.ts — same plaintext vote encrypts differently every time
encrypt(message: bigint): ElGamalEnvelope {
  // 2048-bit FFDHE prime group — RFC 7919 (same group as modern TLS)
  const { prime: p, generator: g } = getGroup(2048);

  // Random ephemeral key k — this is why two identical votes look different
  const k = randomBigInt(2n, p - 2n);

  const c1 = modPow(g, k, p);                        // g^k mod p
  const c2 = (message * modPow(publicKey, k, p)) % p; // m * h^k mod p

  return { v: 1, c1: c1.toString(16), c2: c2.toString(16) };
}

Why 2048-bit? Breaking this requires factoring a 617-digit number. The world's fastest supercomputer would need longer than the age of the universe.

Three-state verification pipeline

backend/src/services/vote.service.ts — verifyVote() 
// vote.service.ts
async verifyVote(serialNumber: string): Promise<VerifyVoteResult> {
  const vote = await voteRepository.findBySerial(serialNumber);

  // Step 1 — recompute hash from stored ciphertext
  const recomputedHash = encryptionService.hashEncryptedData(vote.encryptedData);
  const hashValid = recomputedHash === vote.voteHash;

  // Step 2 — check blockchain anchor
  const blockchainRecord = await blockchainService.getVoteRecord(serialNumber);
  const blockchainConfirmed = blockchainRecord?.voteHash === vote.voteHash;

  return {
    verified: hashValid && blockchainConfirmed, // green checkmark
    // OR: hashValid=false → red integrity warning
    // OR: vote.status='SUPERSEDED' → amber superseded notice
  };
}

Milestone 5 — Dual-PIN Distress System Week 5

Rationale Section 10 of Kenya's Elections Offences Act (2016) criminalises voter coercion — but criminalising it and preventing it are different things. Traditional voting offers zero technical protection once a voter is physically threatened. My system gives a coerced voter a secret weapon: a second PIN that looks and feels identical to the real one from the outside, but silently flags the vote and alerts IEBC. Combined with revoting, the coerced voter has two independent paths to cast a genuine ballot.

What was built

  • Normal PIN: voter-chosen 4 digits, validated against sequential/repeating patterns
  • Distress PIN: server-generated, guaranteed to differ in at least 2 digit positions
  • Both hashed independently with Argon2id — GPU-resistant, memory-hard
  • Distress PIN delivered to voter via SMS or email — only the voter knows both
  • Entering distress PIN silently sets isDistressFlagged = true on the vote

Generating a guaranteed-different distress PIN

backend/src/services/voter.service.ts — setVoterPin() 
// voter.service.ts
let distressPin: string;
do {
  distressPin = Array.from({ length: 4 },
    () => randomInt(0, 10)).join('');

  // Must differ from normal PIN in at least 2 digit positions
  const diffPositions = distressPin.split('')
    .filter((d, i) => d !== pin[i]).length;

  if (diffPositions >= 2 && !isAllSame && !isSequential) break;
} while (attempts++ < 100);

// Both hashed independently — one hash cannot derive the other
const normalPinHash   = await argon2.hash(pin,        { type: argon2.argon2id });
const distressPinHash = await argon2.hash(distressPin, { type: argon2.argon2id });

Silent detection at vote cast time

backend/src/services/vote.service.ts — castVote(), PIN check block 
// vote.service.ts — voter receives a normal receipt either way
let isDistressVote = false;
let pinValid = await argon2.verify(voterRecord.normalPinHash, input.pin);

Milestone 6 — Password Auth & OTP Login Week 6

Rationale A voting system that only biometric users can access disenfranchises every voter without a modern device. I built three independent authentication methods — password, OTP, and biometric — because different Kenyan voters have different capabilities. The OTP path uses Africa's Talking, a Kenyan company, for SMS delivery — keeping voter data within the continent and reducing foreign dependency.

What was built

  • POST /api/auth/login — password-based login returning JWT
  • POST /api/auth/request-otp and /api/auth/verify-otp — OTP login
  • /register — voter self-registration with live password validation
  • OTP: 6-digit, 10-minute TTL, hashed in Redis, purpose-scoped
  • Africa's Talking SMS sandbox integration
  • Setup JWT — short-lived token issued post-KYC for PIN/biometric enrollment only

OTP with Redis TTL and purpose scoping

backend/src/services/otp.service.ts 
// otp.service.ts
async requestOtp(nationalId: string, purpose: 'LOGIN' | 'CONTACT_VERIFY') {
  const code = String(randomInt(100000, 999999)); // 6-digit

  // Scoped key prevents cross-purpose replay
  const key = `otp:${purpose}:${nationalId}`;
  await redis.set(key, await argon2.hash(code), 'EX', 600); // 10-min TTL

  await notificationService.sendOtp({ nationalId, code, purpose });
}

Milestone 7 — WebAuthn Biometric Authentication Week 7

Rationale Passwords can be stolen. OTPs can be intercepted. WebAuthn/FIDO2 — the same technology behind Google and Apple Passkeys — cannot be phished because the cryptographic challenge is bound to the exact origin. Critically, I store only a public key on the server. The raw fingerprint never leaves the device and never crosses the network. A database breach cannot expose voter biometrics.

What was built

  • Backend: @simplewebauthn/server — challenge generation and assertion verification
  • Frontend: @simplewebauthn/browser — triggers native browser biometric prompt
  • Credential stored per voter — supports multiple enrolled devices
  • Enrollment integrated into registration flow and admin approval workflow

Milestone 8 — KYC & UX Hardening Week 8

Rationale Three specific problems closed the gap between prototype and production quality. First, Persona KYC was opening in a new tab — voters were closing the window. I embedded it as an inline iframe with postMessage auto-advance. Second, the country code selector wasn't showing flag emojis on Windows Chrome — a documented OS limitation where the system renderer ignores emoji in native select elements. I replaced it with a custom React component. Third, error messages for wrong passwords and PINs were generic; I added shake animations and inline placement.

What was built

  • Persona inline iframe with postMessage auto-advance on approval
  • 3-attempt tracking — fail 3 times → routed to in-person appointment
  • Custom CountryCodeSelect component — all 195 countries with flag emoji + search
  • Shake animation on wrong PIN/password entries
  • Red borders and inline error placement on all auth forms

A1 Sectoral Resilience

VeriVote directly addresses the most critical security gap in Kenya's electoral sector: the absence of a tamper-evident, independently-verifiable voting record.

VeriVote's blockchain anchoring makes this attack impossible — any tampering with a vote record is immediately detectable by the public /verify endpoint without any privileged access.

Specific gap closed: From zero voter-accessible verification to cryptographic, blockchain-backed proof available to every citizen with an internet connection.

  • Three independent login methods mean a voter is never locked out by a single point of failure.
B1 Originality & Sophistication

Two original contributions not found in existing e-voting literature I reviewed:

  1. Dual-PIN coercion protection with guaranteed distance — the distress PIN is algorithmically guaranteed to differ in at least 2 digit positions from the normal PIN, preventing accidental triggering. The detection is silent and architecturally invisible to an attacker watching the screen.
  2. ElGamal probabilistic encryption with SHA-256 hash anchoring — the probabilistic property means two voters who made identical selections produce different ciphertexts. A mass comparison attack to reconstruct vote distribution from ciphertexts is computationally infeasible. The hash independently verifies integrity without requiring decryption.

The WebAuthn biometric integration is also non-trivial — it required implementing the full FIDO2 challenge-response protocol with Redis-backed challenge storage, origin validation, and per-voter credential management.

B2 System Performance & Reliability
  • Vote casting: single async operation — voter receives receipt immediately while blockchain anchoring runs in the background.
  • Rate limiting per endpoint: 5 req/15 min for auth, 5 for vote casting, 20 for verification, 200 for admin — all enforced by Redis.
  • Blockchain unavailability is non-fatal — the API returns the vote receipt immediately; blockchain confirmation updates asynchronously.
  • OTP TTL enforced at the Redis layer — no clock-skew vulnerability.
  • Argon2id hashing is deliberately compute-intensive (memory-hard) — this is a feature, not a bug, as it makes brute-force attacks impractical even with GPU farms.

Threshold ElGamal — Decentralised Vote Tallying

Currently, the ElGamal private key is held by the backend server. In a production national election, no single person should be able to decrypt all votes. Threshold ElGamal splits the private key across multiple IEBC officials.

How it works

  1. At election setup, the private key is split into n shares using Shamir's Secret Sharing (e.g., 5-of-9: any 5 of 9 commissioners can decrypt).
  2. Each share is given to one IEBC commissioner — no single person holds the full key.
  3. After voting closes, a decryption ceremony requires at least 5 commissioners to submit their share to a secure API endpoint.
  4. The system reconstructs the private key in-memory (never persisted), decrypts all votes, produces the tally, and immediately discards the key.
  5. The tally result is cryptographically signed by the combining service and published to the blockchain.
